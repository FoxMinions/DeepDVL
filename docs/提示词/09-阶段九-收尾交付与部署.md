# 09 · 阶段九：收尾交付与部署

## 本阶段目标

把项目**收口交付**：补全 README、验证生产 build、整理 GitHub 仓库、（可选）部署到 GitHub Pages 让大屏在线可访问。本阶段结束时，DeepDVL 是一个对外体面的开源项目——任何人 clone 下来能跑、能看懂、能在线预览。

## 前置依赖

- 阶段八已完成：单测 + E2E + lint 全绿。

## 为什么这么做

**为什么要认真收尾**：前八阶段都在「造」，但开源项目的「门面」决定别人愿不愿看。README 是第一印象，build 产物验证是「真能跑」的证明，在线预览是「能直接看」的诚意。

**README 要写什么**：不是流水账，而是回答访客三个问题——这是啥？我能学到啥？怎么跑起来？加上截图位、技术栈、阶段进度、目录结构、部署链接。

**GitHub Pages 部署**：大屏是纯前端静态产物，天然适合 GitHub Pages。`npm run build` 产 `dist/`，用 GitHub Actions 自动部署，`main` 分支推送即上线。这是开源前端项目的标准动作，也是教学项目「看得见的成果」。

**教学要点**：理解「**交付是开发的一部分**」——能跑、能看、能被别人复现，才算真完成。很多教程止步于「我本地能跑」，但生产 build 与部署会暴露开发时没遇到的问题（路径、base、环境变量）。

## 要落地的产物

- 根 `README.md`：在现有基础上补全——项目简介、截图位（占位，后续补）、技术栈表、9 阶段进度（勾选已完成）、目录结构、本地运行、部署链接（部署后补）、协议。保留已有内容，**追加/完善**而非重写。
- `npm run build` 验证：`dist/` 产出，`vite preview` 本地预览正常。
- `vite.config.ts`：若部署到 GitHub Pages 子路径（`https://用户名.github.io/DeepDVL`），需设 `base: '/DeepDVL/'`。
- `.github/workflows/deploy.yml`（可选）：GitHub Actions，`main` 推送时 `npm ci && npm run build` → 部署 `dist/` 到 Pages。
- GitHub 仓库整理：确认 `About` 描述、Topics（`data-visualization` `big-screen` `vue3` `echarts` `teaching` 等）、README 在网页渲染正常。
- 仓库标签/Release（可选）：打 `v0.1.0` tag 标记首个完整版本。

## 思考题 / 进阶

1. 为什么 GitHub Pages 部署要设 `base: '/DeepDVL/'`？不设会怎样？（提示：资源路径会指向根，404）
2. `npm ci` 与 `npm install` 的区别？为什么 CI 里用 `ci`？（提示：lockfile 严格、更快、可复现）
3. 如果大屏要部署到自己的域名或子路径，`base` 该怎么改？
