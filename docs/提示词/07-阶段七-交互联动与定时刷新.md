# 07 · 阶段七：交互联动与定时刷新

## 本阶段目标

把分散的图表用 **Pinia store** 统一驱动，实现**定时刷新**与**错误降级 UI**。本阶段结束时，数据从「组件各自 fetch」升级为「store 统一管理」，页面每 N 秒自动刷新，且数据加载失败时整屏不会白屏而是显示降级提示。

## 前置依赖

- 阶段六已完成：DataHubChart 与三栏布局就绪。
- 已安装 `pinia`（阶段三/四预装；若没装，本阶段装）。

## 为什么这么做

**为什么要 store**：前几个阶段每个组件自己调 `fetchDashboardData()`，问题：
1. 多个组件各 fetch 一次，请求重复。
2. 加载/错误状态散在各组件，无法统一处理。
3. 后续想做「点击某节点联动其他图表」，没有全局状态就没法联动。

用 Pinia 把 `data / loading / error / loadDashboard()` 收口，组件只从 store 读——**单一数据源**，刷新与联动都顺理成章。

**为什么要定时刷新**：数据大屏常驻展示，静态数据显得「死」。定时刷新让大屏「活」起来，也是真实大屏的标配。但要处理两个坑：
1. **无闪烁刷新**：`setOption` 是 diff 式更新，数据变只重绘变化部分，不重建 canvas——所以定时刷新不会闪。前提是图表组件正确用了阶段四的范式（不重新 init）。
2. **定时器泄漏**：`setInterval` 必须在组件卸载时 `clearInterval`，否则切路由后还在偷偷请求。

**为什么强调错误降级**：大屏现场展示，后端偶尔抽风。如果请求失败整屏白屏，演示直接翻车。正确做法：catch 错误 → `error` 状态 → UI 显示「数据加载失败，X 秒后重试」+ 保留上一次成功的数据继续展示（如果有）。这是大屏的**生产级健壮性**，也是教学重点。

**教学要点**：理解「**状态机**」思维——大屏有 `loading / success / error` 三态，UI 按状态分支渲染，而不是「有数据就显示没数据就空」。

## 要落地的产物

- `src/stores/dashboardStore.ts`：Pinia store，state `{ data, loading, error }`，action `loadDashboard()`（调 `fetchDashboardData`，try/catch/finally，成功存 data 清 error，失败存 error 并 logger.error）。
- `src/app/main.ts`：挂载 `createPinia()`。
- `src/views/DashboardView.vue`：从 store 读数据（替换之前的临时 ref + fetch），`onMounted` 调 `loadDashboard()`，并 `setInterval` 每 30 秒刷新一次，`onBeforeUnmount` 清定时器。
- 错误降级 UI：`error` 非空时显示「数据加载失败，将在 N 秒后重试」占位，不白屏；若已有旧 data 可选择保留显示 + 顶部小提示。
- 图表联动（轻量）：点击 `DataHubChart` 某节点时，在 `ScreenHeader` 或某处显示该节点详情（用 store 暂存 `selectedNodeId`，组件间通过 store 联动）。本项可轻量实现，证明联动可行即可。
- `src/logs/logger.ts`：在 store 的 catch 里用 `logger.error` 记录失败。

## 思考题 / 进阶

1. 为什么定时刷新用 `setInterval` 而不是 `setTimeout` 递归？（提示：前者间隔固定但可能叠加，后者能避免上一次未完成就开始下一次——大屏该选哪个？）
2. 如果刷新一次要 5 秒，而间隔是 3 秒，会发生什么？该怎么防？（提示：上一次未完成就被下一次打断，可加 loading 守卫）
3. 错误降级时「保留旧数据」和「显示失败占位」哪种更好？（提示：看场景——演示时保留旧数据更稳，开发时显式失败更易调试）
4. 进阶：把刷新间隔做成可配置（env 变量 `VITE_REFRESH_INTERVAL`），并加一个「暂停刷新」按钮（大屏演示时常用）。
