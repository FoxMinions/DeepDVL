# 08 · 阶段八：测试体系与代码质量

## 本阶段目标

搭起**单元测试 + E2E 测试**双保险，并让 `npm run test && test:e2e && lint` 全部全绿。本阶段结束时，项目有了可回归的测试保护网，后续改动有信心不破坏既有功能。

## 前置依赖

- 阶段七已完成：store 驱动、定时刷新、错误降级就绪。
- 项目至此已有：折线图、指标卡片、中心 DataHub、三栏布局、store。

## 为什么这么做

**为什么写测试**：教学项目常跳过测试，但大屏项目尤其需要——
1. `format` 这种纯函数，不测很容易在边界值（负数、0、小数）翻车。
2. `dashboardService` 的 mock/api 分支，没测试保护，重构时容易改坏。
3. `dataSource` 的环境变量切换，是「悄悄坏掉」的高发区。

**单测选 Vitest**：Vite 原生，配置零成本，与项目同源。测重点：纯函数（format）、service 分支（mock/api）、dataSource 切换、logger 输出。

**E2E 选 Playwright**：大屏最终要「人眼看得见」地对，单测管不到布局/渲染。Playwright 跑真浏览器，验证：页面不空白、标题在、卡片在、图表容器在、无滚动条。这是大屏的「端到端验收」。

**测试金字塔**：单测多（快、便宜）、E2E 少（慢、贵但真实）。教学项目里 E2E 不必覆盖所有交互，只跑核心「大屏能看」流程。

**教学要点**：理解「**测试不是证明程序对，而是防止程序退化**」——每次改动后跑一遍，绿的就有信心。这是工程化心态的核心。

## 要落地的产物

- 安装 `vitest`、`@vue/test-utils`、`jsdom`、`@playwright/test`、`playwright`。
- `vite.config.ts`：加 `test` 配置（environment: jsdom），或单独 `vitest.config.ts`。
- `package.json` scripts 补：`test: vitest run`、`test:e2e: playwright test`。
- 单元测试：
  - `src/tests/unit/format.test.ts`：测 `formatNumber`（千分位、0、负数）、`formatTrend`（正负、0、小数）。
  - `src/tests/unit/dashboardService.test.ts`：mock 模式返回数据、api 模式走 httpClient（mock axios）。
  - `src/tests/unit/dataSource.test.ts`：`VITE_DATA_SOURCE=api` → 'api'，其他 → 'mock'。
  - `src/tests/unit/logger.test.ts`：各级别调用 console 对应方法（spy console）。
  - `src/tests/unit/resize.test.ts`（可选）：`observeResize` 调用后 observer.observe 被调用。
- E2E：
  - `playwright.config.ts`：baseURL `http://localhost:5173`，webServer 跑 `npm run dev`。
  - `src/tests/e2e/dashboard.spec.ts`：打开首页 → 验证标题存在 → 至少一个 `metric-card` 存在 → 至少一个图表 `data-testid` 容器存在 → 无横向滚动条（`document.body.scrollWidth <= window.innerWidth`）。
- 配置文件：`eslint.config.ts` 视情况纳入 test 文件规则。

## 思考题 / 进阶

1. 测 `dashboardService` 的 api 分支时为什么要 mock axios 而不是真发请求？（提示：单测要快、要隔离、不能依赖网络）
2. E2E 为什么只测「大屏能看」而不测具体数值？（提示：数值来自 mock 会变；E2E 验证结构与可见性，数值正确性归单测）
3. `dataSource.test.ts` 测环境变量切换时，如何在不污染其他测试的前提下临时改 `import.meta.env`？（提示：vi.stubEnv + afterEach restore）
4. 进阶：给 DataHubChart 加一个「数据为空时显示空状态」的 E2E 用例——把 mock 改成返回空数组，验证不报错、显示空态。
